# LCORE-D Adaptive Feature Path

This note records the first repository-level data-source switch from the office FortiGate trace to LCORE-D as the main benchmark-oriented data path.

## Source

- Dataset: `LCORE-D: A Benchmark Dataset for Core Network Analysis`
- URL: `https://data.mendeley.com/datasets/77sztrg5ks/2`
- DOI: `10.17632/77sztrg5ks.2`
- Published version: Version 2, published 2025-12-17
- Stated scope: 407 hours of fault-annotated ISP core-network monitoring data from a GNS3-emulated network.
- Working fault labels for benchmark comparison: single link failure, multiple link failure, misconfiguration, routing misconfiguration, line card failure, ICMP blocked by firewall, node failure, multiple nodes failures, single node failure, and SNMP agent failure.

The public page is accessible, and the browser-facing anonymous API exposes the file manifest:

`https://data.mendeley.com/public-api/datasets/77sztrg5ks/files?folder_id=root&version=2`

The lower-level `api.data.mendeley.com` API may return `401` without credentials, so the edge runtime uses the public manifest/download URLs observed from the dataset page.

1. Download or mount the dataset files locally.
2. Run the adaptive feature preparer against the local file, directory, or zip archive.

## Why This Module Exists

The original runtime path assumes a fixed FortiGate parser contract:

`vendor log -> FortiGate parser -> 66-field structured fact -> deterministic correlator`

LCORE-D is a broader benchmark dataset with topology and fault annotations. A fixed FortiGate KV parser is the wrong abstraction for it, so the new path adds a schema discovery layer before the existing core pipeline:

`LCORE-D files -> adaptive feature plan -> canonical NetOps fact JSONL -> existing forwarder/correlator`

## Command

```bash
python3 -m core.benchmark.lcore_adaptive_prepare \
  --input /data/netops-runtime/LCORE-D/raw \
  --output-jsonl /data/netops-runtime/LCORE-D/output/events-lcore-d.jsonl \
  --plan-json /data/netops-runtime/LCORE-D/work/feature-plan.json
```

For live edge replay, append events gradually so the existing `edge-forwarder` controls Kafka delivery from a normal JSONL source:

```bash
python3 -m edge.lcore_streamer.main \
  --input /data/netops-runtime/LCORE-D/raw \
  --output-jsonl /data/netops-runtime/LCORE-D/output/events-lcore-d.jsonl \
  --plan-json /data/netops-runtime/LCORE-D/work/feature-plan.json \
  --checkpoint-json /data/netops-runtime/LCORE-D/work/stream-checkpoint.json \
  --events-per-second 20
```

Supported inputs:

- CSV files
- JSONL / NDJSON files
- JSON arrays or objects with `records`, `rows`, or `data`
- Zip archives containing the formats above
- Directories containing any supported file type

## Adaptive Feature Groups

The preparer samples rows and separates fields into:

- `primary_time_field`: timestamp source for `event_ts`
- `label_fields`: fault/scenario/anomaly/status fields
- `entity_fields`: node/router/device/interface/link identity fields
- `topology_fields`: source/destination/link/path/neighbor/site fields
- `metric_fields`: numeric monitoring features
- `categorical_fields`: low-cardinality non-numeric context
- `scenario_values`: normalized abnormal scenario names observed in labels. LCORE-D labels are preserved as ten comparable scenarios instead of being collapsed into coarse families.

The downloaded version-2 CSV files currently expose a compact row label column named `class` with values such as `H`, `F`, `T`, and `TH`. The adapter preserves the original `label_value` and maps those compact labels as follows for runtime behavior:

- `H`: `healthy`
- `F`: `induced_fault`
- `T`: `transient_fault`
- `TH`: `transient_healthy`

If a local LCORE-D export or future version provides explicit scenario labels, the ten scenario names below are still preserved one-to-one.

Each output row becomes a canonical fact with:

- `event_id`
- `event_ts`
- `type=telemetry`
- `subtype=fault_annotation` when an abnormal annotation is present
- `fault_context`
- `topology_context`
- `device_profile`
- `change_context`
- `dataset_context`
- `feature_vector`

## Correlator Integration

The correlator now supports `annotated_fault_v1` in addition to the existing FortiGate-oriented rules:

- `deny_burst_v1`
- `bytes_spike_v1`
- `annotated_fault_v1`

`annotated_fault_v1` only triggers when an event carries a fault annotation, such as the `fault_context` generated by the adaptive feature module. It emits on fault-state transitions per entity/scenario instead of firing on every abnormal row, which keeps benchmark replay volume bounded.

The normalized LCORE-D scenario names are:

- `single_link_failure`
- `multiple_link_failure`
- `misconfiguration`
- `routing_misconfiguration`
- `line_card_failure`
- `icmp_blocked_firewall`
- `node_failure`
- `multiple_nodes_failures`
- `single_node_failure`
- `snmp_agent_failure`

## Current Boundary

This module does not yet implement topology-aware subgraph extraction. It prepares the data contract needed for that next step by preserving topology fields, path signatures, neighbor references, fault scenarios, and numeric feature vectors in a stable canonical event shape.
