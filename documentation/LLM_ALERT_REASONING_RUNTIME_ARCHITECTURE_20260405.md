# LLM Alert Reasoning Runtime Architecture / LLM 告警推理运行时架构说明（2026-04-05）

本文档定义当前分支中的 LLM downstream path。起点是 `netops.alerts.v1`。输入对象是已成立的 alert contract 和 cluster trigger。输出对象是结构化 suggestion、候选事件图、调查会话、推理追踪和 runbook 计划。边缘解析、事实转发、规则判定、JSONL 审计和 ClickHouse 存储继续沿现有主链运行。当前代码已经把 `reasoning_runtime_seed` 挂到 `evidence_bundle`，并把 `candidate_event_graph_id`、`investigation_session_id`、`reasoning_trace_id` 和 `runbook_plan_outline` 投影到 suggestion。

> [!IMPORTANT]
> `core/correlator` 负责第一判定。`core/aiops_agent` 负责 alert downstream 推理与建议。

> [!WARNING]
> `192.168.1.23` 仅适合边缘采集与转发。该节点为 `Intel Xeon E3-1220 v5`、`4` 个 CPU 线程、`7.7 GiB` 内存、`914 GiB` 根分区。`192.168.1.27` 负责核心平面。该节点为 `Intel Xeon Silver 4310`、`24` 个 CPU 线程、`14 GiB` 内存、`1.8 TiB` 根分区。当前代码路径不在任一节点上启用本地 GPU 推理。

## 机制

当前系统里的 downstream path 采用两段结构。第一段是 deterministic candidate graph construction。输入来自 alert contract、cluster aggregator、ClickHouse history lookup 和现有 context 字段。输出是一张带有候选节点和候选关系的局部事件图。第二段是 explanation-oriented reasoning。这个阶段当前还在 foundation 期。后续状态机将围绕 `ClusterSelectOrRefine -> ContextAssemble -> EvidencePackBuild -> HypothesisGenerate -> HypothesisCritique -> RunbookRetrieve -> RunbookDraft -> RunbookReview -> SuggestionProjection` 展开。状态机外部保留 `human_gate`。回退边保留 `cluster_too_broad`、`evidence_insufficient`、`hypothesis_conflict`、`runbook_not_applicable`、`runbook_overreach` 和 `high_risk_or_missing_approval`。

当前 context engineering 采用对象约束方式推进。`EvidencePack V2` 现在已经显式拆出 `direct_evidence`、`supporting_evidence`、`contradictory_evidence`、`missing_evidence`、`freshness`、`source_reliability` 和 `lineage`。phase router 按阶段分配这份对象的视图。`HypothesisGenerate` 当前读取 direct/supporting/contradictory/missing 证据，再读取 rule/path/device 上下文。`HypothesisCritique` 当前读取 direct/supporting/contradictory 证据，再读取 policy/sample/change 上下文。`RunbookRetrieve` 和 `RunbookDraft` 当前读取 direct/supporting/missing 证据，再读取 device/policy/change 和 `runbook_plan_outline`。`RunbookReview` 当前读取 direct/contradictory/missing 证据，再读取 change、历史摘要和审批边界。当前 suggestion payload 已经带上 `hypothesis_set`、`review_verdict`、`runbook_draft` 和 `reasoning_stage_requests`。前端 projector 的 inference / action 节点和 convergence field 已优先读取前三个对象。`reasoning_stage_requests` 当前保留给 audit、replay 和 future remote execution。旧 runtime suggestion 不带这些对象时，gateway 会生成兼容 fallback。working memory 保留 `accepted_hypotheses`、`rejected_hypotheses`、`unresolved_questions`、`missing_evidence`、`pending_checks` 和 `risk_flags`。context compression 压缩结构化摘要。answer contract checker 在最终投影前检查证据引用、事实与推断分离、缺失信息、审批边界和回滚边界。

runbook 路径已在当前架构中预留正式位置。当前对象是 `runbook_plan_outline`。后续对象是 `RunbookPlan`。目标字段包括 applicability、evidence refs、prechecks、diagnostic steps、recommended actions、risk level、blast radius、approval required、rollback prerequisites、rollback steps 和 missing information。当前分支不引入自动执行。当前分支不引入设备写回。工具层保留 allowlist、参数校验、超时、结果结构化和调用审计。

## 模型服务接口

模型接入继续沿用现有 provider 抽象。`template` provider 保持当前路径。外部模型服务通过 `http`、`gpu_http` 或 `external_model_service` provider 标识接入。配置项已经在 `AgentConfig` 中显式化，包括 `provider_compute_target` 和 `provider_max_parallelism`。当前默认约束是 `provider_compute_target=external_gpu_service`、`provider_max_parallelism=1`，由核心节点发起请求。当前代码里 `provider_routing.py` 会生成结构化 routing hint，包含 `compute_target`、`max_parallelism`、`request_kind`、`suggestion_scope`、`candidate_event_graph_id`、`investigation_session_id` 和 `runbook_plan_id`。`reasoning_stage_requests.py` 已经把这个 routing hint 和 `hypothesis_critique`、`runbook_draft` 的 `phase_context` 组装成下一步可直接发给远端模型服务的 request contract。当前分支尚未执行这一步远端调用。

边缘节点不承载模型执行。核心节点承载 suggestion、evidence、session 和 future orchestration。当前核心节点内存为 `14 GiB`。这一预算适合模板 provider、本地结构化对象构造、阶段感知上下文裁剪和远端模型转发。它不适合和当前 Kafka consumer、ClickHouse 查询、gateway 进程一起承载大模型。GraphRAG 目前只规划给非结构化知识面。运行时对象和当前热路径不依赖完整 GraphRAG indexing。

当前仓库里已经存在几组与这个 path 直接相关的参数。`AIOPS_CLUSTER_WINDOW_SEC`、`AIOPS_CLUSTER_MIN_ALERTS` 和 `AIOPS_CLUSTER_COOLDOWN_SEC` 控制 cluster trigger 的窗口、触发阈值和冷却时间。`AIOPS_PROVIDER`、`AIOPS_PROVIDER_ENDPOINT_URL`、`AIOPS_PROVIDER_MODEL`、`AIOPS_PROVIDER_TIMEOUT_SEC`、`AIOPS_PROVIDER_COMPUTE_TARGET` 和 `AIOPS_PROVIDER_MAX_PARALLELISM` 控制 provider 路径。`CLICKHOUSE_*` 参数控制历史检索来源。当前挂载 runtime 中有 `691` 个 alert JSONL 文件和 `201003` 条 alert，时间范围是 `2026-03-04T15:09:11+00:00` 到 `2026-04-02T16:23:04+00:00`。当前挂载 runtime 中有 `603` 个 suggestion JSONL 文件和 `222023` 条 suggestion，时间范围是 `2026-03-09T05:08:56.549849+00:00` 到 `2026-04-05T18:03:18.303384+00:00`。最近 24 个 alert 分桶中有 `1452` 条 `deny_burst_v1|warning` 和 `1` 条 `bytes_spike_v1|critical`。最近 24 个 suggestion 分桶中有 `1690` 条 `alert` 建议和 `32` 条 `cluster` 建议。

## 最终形态与完成状态细化说明

本节把当前代码已经落地的部分和项目最终目标态放到同一个坐标系里。本文档的“最终形态”指的是这个项目完成后在仓库结构、运行路径、对象契约、部署职责、实验面和论文表达上的稳定终局。本文档的“完成状态”指的是截至当前分支已经真正进入代码、已经经过测试、已经进入 suggestion payload 或 frontend projection 的部分。当前分支的整体完成度已经越过“概念设计”阶段，进入“本地结构化链路基本闭合、只差真实远端模型执行”的状态。当前仓库并不存在第二条和主链竞争的 AI 处理线。边缘平面、核心判定面、双持久化面、前端投影面都继续沿既有路径工作。新增的内容全部位于 `core/aiops_agent` 内部，作为 `netops.alerts.v1` 之后的增强层存在。这个增强层当前已经固定的基本事实是：上游 `fortigate-ingest` 负责厂商日志抽取和 fact 规范化；`edge_forwarder` 负责结构化 fact 转发；`core/correlator` 负责 alert 成立性判断；`alerts_sink` 和 `alerts_store` 负责审计与查询；`cluster_aggregator` 负责 same-key repeated-pattern 触发；`aiops_agent` 负责 alert downstream 证据组装、结构化 suggestion、结构化 reasoning object 投影和未来模型阶段请求装配。项目最终形态不会改变这个主次关系。项目不会把 raw syslog 直接交给模型去决定告警是否成立。项目不会让模型回头修正 correlator 的第一判定。项目也不会在没有审批和回滚面之前打通设备写回。

项目最终的代码结构可以按四组理解。第一组是 deterministic plane，位于 `edge/`、`core/correlator`、`core/alerts_sink`、`core/alerts_store` 和现有 `cluster_aggregator`。第二组是 alert downstream reasoning plane，位于 `core/aiops_agent` 及其子目录。第三组是 runtime projection plane，位于 `frontend/gateway` 和 `frontend/src`。第四组是 future controlled execution plane，目前只保留接口位置和审批边界，不在当前交付路径里。当前已经完成的 reasoning plane 内部模块包括 `evidence_bundle.py`、`evidence_pack_v2.py`、`hypothesis_set.py`、`review_verdict.py`、`runbook_draft.py`、`provider_routing.py`、`reasoning_stage_requests.py` 和 `alert_reasoning_runtime/` 目录下的 `candidate_event_graph.py`、`investigation_session.py`、`reasoning_trace.py`、`runbook_plan_outline.py`、`rule_based_seed_adapter.py`、`phase_context_router.py`。这些模块已经形成一个稳定层级。`evidence_bundle.py` 仍是 alert-side context 的第一聚合点。`evidence_pack_v2.py` 负责把 heterogeneous context 收束成稳定证据对象。`hypothesis_set.py` 负责把 provider 输出和 Evidence Pack 合成为结构化假设集。`review_verdict.py` 负责把 hypothesis 和 evidence 再做一次边界审查。`runbook_draft.py` 负责把受限 runbook planning surface 变成真正可以被前端消费的结构化 planning output。`provider_routing.py` 负责把 provider 选择、计算目标和阶段路由约束写成显式 hint。`reasoning_stage_requests.py` 则是当前分支里最接近下一阶段真实模型执行的一层，它已经把 `hypothesis_critique` 和 `runbook_draft` 变成结构化的远端请求契约。

项目最终的运行路径已经可以明确到对象级。对于 alert-scope，请求链是 `alert -> build_alert_history_context -> build_alert_evidence_bundle -> build_alert_inference_request -> provider inference -> build_alert_pipeline_suggestion -> gateway projection -> frontend runtime console`。在这条链里，`build_alert_evidence_bundle` 已经同时挂上普通 evidence bundle、`reasoning_runtime_seed` 和 `evidence_pack_v2`。`build_alert_pipeline_suggestion` 已经在 suggestion payload 中写入 `hypothesis_set`、`review_verdict`、`runbook_draft` 和 `reasoning_stage_requests`。对于 cluster-scope，请求链是 `alert stream -> cluster_aggregator.observe -> cluster trigger -> build_cluster_evidence_bundle -> build_cluster_inference_request -> provider inference -> build_pipeline_suggestion -> gateway projection -> frontend runtime console`。当前 cluster 路径已经完全复用同一套 structured reasoning contract，只是 seed graph、session scope 和历史上下文字段不同。项目最终形态也会保持这一点：alert-scope 和 cluster-scope 共享对象层、共享阶段矩阵、共享 provider routing，只在 seed 和 context 上分化，不会各自维护一套完全不同的推理系统。

项目最终的对象关系已经基本定型。`reasoning_runtime_seed` 是从 rule-based world 进入 reasoning world 的第一个桥对象。它由 `rule_based_seed_adapter.py` 构造，当前包含 `candidate_event_graph`、`investigation_session`、`reasoning_trace_seed` 和 `runbook_plan_outline`。`candidate_event_graph` 代表一个局部候选事件图，不是全局知识图谱。它用于回答“围绕当前 alert 或 cluster，系统此刻有哪些可解释的候选关系”。`investigation_session` 代表一次调查会话，不是长对话 memory。它用于回答“这次调查是从哪里进入的，目前在哪个阶段，哪些节点已被选中，哪些 retriever 应该参与”。`reasoning_trace_seed` 代表一个 trace 起点，用于回答“当前链路理论上应该经历哪些阶段”。`runbook_plan_outline` 代表 rule-based side 提前准备的 planning seed，用于回答“这次 planning 的适用边界、预检查、人工审批边界和回滚草案是什么”。这四个对象当前都已经实际进入 suggestion payload 所依赖的 evidence path，而不是只停留在文档草图里。

Evidence Pack V2 是整个项目最终形态里最重要的稳定输入对象。它当前已经固定字段，不再建议继续频繁改名或改层级。它的核心作用是把以后所有阶段都必须共享的证据对象定死。`direct_evidence` 记录直接成立的事实，例如 `alert.rule_id`、`alert.severity`、`topology.service`、`topology.src_device_key`、`path.path_signature`、`rule.metrics`。`supporting_evidence` 记录能加强判断但不构成第一判定的上下文，例如 `recent_similar_1h`、`cluster_size`、`recent_alert_samples`、`neighbor_refs`、`known_services`、`change_refs`、`policy hits`。`contradictory_evidence` 记录削弱假设的事实，例如 cluster gate 未满足、history 为零、没有 change signal、与设备 profile 不一致。`missing_evidence` 记录当前应该存在但尚未提供的字段，例如 path signature 缺失、change records 缺失、policy identity 缺失。`freshness` 用于保留时间上的可用性判断。`source_reliability` 用于区分 fact-side、history-side、change-side 和 future runbook-side 的来源可信度。`lineage` 用于把 evidence ref 与 source section、source field、source ref 绑死。项目最终形态里，后续任何新的 reasoning stage 如果不能直接消费 `Evidence Pack V2`，就应当被视为对象边界设计失败。

HypothesisSet、ReviewVerdict、RunbookDraft 已经构成当前完成态的三段结构化 reasoning output。`HypothesisSet` 当前已经能把 provider 返回的原始 hypothesis 列表变成带主假设、排序、支持证据引用、反证证据引用、缺失证据引用、下一步建议和 review state 的对象。它解决的是“推理结论不能只有几句 summary 文本”这个问题。`ReviewVerdict` 当前已经能把 hypothesis 和 evidence 再做一次面向边界的审查，输出 evidence sufficiency、temporal freshness、topology consistency、overreach risk、remediation executability、rollback readiness 六个固定检查项，并给出 `recommended_disposition` 和 `approval_required`。它解决的是“建议生成后缺少结构化审查”这个问题。`RunbookDraft` 当前已经能把 deterministic planning seed、ReviewVerdict、HypothesisSet 和受限 recommended action 合成为一个 operator-facing runbook draft，包含 `plan_status`、`title`、`applicability`、`prechecks`、`operator_actions`、`boundaries`、`rollback_guidance`、`approval_boundary`、`evidence_refs` 和 `change_summary`。它解决的是“runbook 不能只是一段自由文本说明”的问题。项目最终形态里，这三个对象将继续保留，而不是被自由文本重新吞掉。

`reasoning_stage_requests.py` 是当前分支里最接近最终远端模型执行的模块。它的意义不是多加一层抽象，而是把“下一步真的要接模型时，到底发什么”先写成强契约。当前它已经生成两个 stage request：`hypothesis_critique` 和 `runbook_draft`。每个 stage request 都带 `stage_request_id`、`parent_request_id`、`stage`、`provider`、`suggestion_scope`、`routing_hint`、`input_contract` 和 `expected_response_schema`。`routing_hint` 来自 `provider_routing.py`。`input_contract` 来自 `phase_context_router.py` 提供的阶段裁剪上下文，再叠加当前已经存在的 reasoning object。`expected_response_schema` 则把远端模型应该返回的 structured output 写死。项目最终形态中，真实模型调用不会直接从 suggestion payload 里随意读字段，也不会直接拿整包 evidence 去自由发挥，而是只能读取这些 stage request。当前停止点也正好落在这里：本地已经具备完整 request contract，下一步只差真实 provider 执行。

项目最终的 provider 分层也已经可以清楚定义。`template` provider 保持 baseline 路径，继续服务当前 deterministic suggestion generation 和所有 fallback 场景。`http`、`gpu_http` 和 `external_model_service` provider 代表未来远端模型调用路径。当前 `AgentConfig` 已经显式支持 `provider_compute_target` 和 `provider_max_parallelism`。当前系统默认把远端 compute target 定位在 `external_gpu_service`，并限制 `max_parallelism=1`。项目最终形态中，核心节点只负责 evidence assembly、stage request serialization、request dispatch、response validation、trace attachment 和 projection。核心节点不会托管本地大模型。边缘节点不会承载任何模型推理。真正模型执行只会发生在外部 GPU 服务或 API-backed provider 上。当前仓库已经为这一步做好 request contract 和 routing hint，尚未真的执行。

前端最终形态也已经有了清晰方向，并且当前完成态已经吃进真实结构化对象。`frontend/gateway/app/runtime_reader.py` 当前已经能够从 suggestion payload 中读取 `hypothesis_set`、`review_verdict` 和 `runbook_draft`，并对旧 suggestion 做兼容 fallback。`frontend/src/types.ts` 当前已经把这些对象作为正式前端类型。`frontend/src/components/liveFlowConsoleEventField.ts` 当前已经优先消费结构化 `runbookDraft`，而不是继续完全靠字符串推荐语句本地拼装。node inspector、convergence field、runtime projector 当前都已经站在“structured reasoning objects 是主产物”的方向上。项目最终形态里，前端将继续强化这一原则：hero area 用来展示当前 incident focus 和系统主舞台；convergence field 用来展示离散事件如何向 hypothesis / incident cluster 收束；runbook inspector surface 用来承接结构化 planning output；普通 summary prose 只作为 projection 辅助，不再作为唯一语义来源。

项目最终的运行数据平面也已经可以界定。`/data/netops-runtime` 继续是 alert JSONL、suggestion JSONL 和 observability 文件的落盘位置。`core/aiops_agent/alert_reasoning_runtime` 继续只是代码目录。这个目录名虽然带 `runtime`，但不代表它保存运行中的数据。它的职责是保存 downstream reasoning 的对象和编排代码。运行时数据仍然由 JSONL sink、ClickHouse 和 gateway 视图负责。项目最终形态里，若后续引入 trace replay 文件、stage request capture、tool call trace 或评测输出，也应当继续落在 runtime 数据目录或单独的 observability 路径中，而不应写回代码目录。

项目最终的控制边界也已经固定。第一条边界是 alert validity boundary。任何 LLM 组件都不能回头决定 alert 是否成立。第二条边界是 execution boundary。当前和后续阶段都不会允许 `runbook_draft` 直接变成设备写回。第三条边界是 approval boundary。所有可能跨出只读诊断面的行为，都必须在 approval-ready control plane 完成前保持停留在建议或审查状态。第四条边界是 rollback boundary。没有 rollback readiness 的计划不得提升为 execution-facing output。第五条边界是 transport boundary。边缘节点不参与任何模型推理和 stage orchestration。第六条边界是 data-contract boundary。远端模型如果接入，只能接收 `reasoning_stage_requests` 这类显式 contract，不得直接读取任意 runtime 文件或直接查询整个仓库上下文。

项目最终的测试面也已经基本成型。当前后端测试已经覆盖 `Evidence Pack V2`、`phase_context_router`、`HypothesisSet`、`ReviewVerdict`、`RunbookDraft`、`reasoning_stage_requests` 和 `run_agent_loop` 在 alert-scope 与 cluster-scope 下的 suggestion emission。当前前端测试已经覆盖 convergence field helper 和 runtime snapshot projection，确保 `runbookDraft`、`hypothesisSet` 和 `reviewVerdict` 这三个对象确实能进入前端模型。项目最终形态中，测试会继续分成四层：对象层单测、service 层单测、gateway projection 层测试、frontend component helper 测试。等真实模型接入后，还需要增加 provider response validation、schema violation fallback、stage replay tests 和离线评测 harness。

项目最终的论文表达已经可以按完成态组织，而不必再用“计划做”描述全部内容。论文里的 baseline 可以直接对应现在这条 deterministic path：`fact normalization -> deterministic alert confirmation -> dual persistence -> template suggestion`。论文里的 enhanced system 可以直接对应现在已经完成的大部分结构化路径：`reasoning_runtime_seed -> Evidence Pack V2 -> HypothesisSet -> ReviewVerdict -> RunbookDraft -> reasoning_stage_requests`。论文里的最后一段 treatment 则是下一步真实模型执行：把 `hypothesis_critique` 和 `runbook_draft` 的 stage request 发给远端模型服务，再回收结构化结果写回 suggestion projection。这样论文实验可以清楚拆出三组：deterministic baseline、structured-object local enhancement、remote model-backed critique/planning enhancement。

截至当前分支，项目的完成状态可以非常具体地表述为：规则基线已完整可运行；alert downstream 的结构化对象层已完成；阶段上下文裁剪已完成；runbook planning output 已完成；未来模型执行请求契约已完成；前端已开始直接消费真实结构化 reasoning object；远端 GPU 或 API-backed provider 尚未真正执行 `hypothesis_critique` 和 `runbook_draft`。这就是当前的明确停止点。下一步如果继续开发，就不再是增加本地 schema 或继续扩文档，而是要做真实 provider execution、response validation、timeout/fallback、trace capture 和 replay/eval 接线。这一步会要求外部 GPU endpoint 或 API provider 凭据。当前文档到这里为止，已经足够支撑架构图绘制、阶段复盘、论文方法章节撰写和后续实现排期。
